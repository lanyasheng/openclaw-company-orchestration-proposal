## 0. 设计摘要（先给结论）

### 一句话结论
**不建议把 LangGraph 或 Temporal 直接替换成 OpenClaw 的“唯一公司级总编排层”。推荐路线是：**

> **P0/P1 先把 OpenClaw 原生编排升级为统一控制平面（session/subagent/acp/watcher/cron/message/browser 之上加 Task Registry + Callback/Event Bus + 统一状态机）；P2 再把 Temporal 作为关键长事务/高 SLA 流程的耐久编排层按需引入；LangGraph 只保留为 agent 内部的认知/规划子图，而不是公司级执行编排内核。**

### 为什么是这条路线
1. **OpenClaw 当前最大缺口不是“不会画图”，而是“状态分散、重试口径不统一、跨运行时审计不足”。**
2. **Temporal 解决的是耐久工作流、重试、定时器、审计与恢复**，适合公司级关键异步流程，但直接全量迁移成本过高。
3. **LangGraph 更擅长 LLM/agent 的思维流与人审断点，不擅长承担公司级异步执行可靠性底座。**
4. **OpenClaw 已经有真实运行时资产**：`sessions_spawn`、subagent runner/watcher、ACP session、callback/watcher bus、cron、message、browser。最经济的路线不是推倒重来，而是先把这些资产规范化、状态化、可观测化。

### 为什么不是另外三个
- **不是纯 LangGraph**：它能编排 agent 思考流，但对跨天任务、精细重试、幂等、跨 runtime 恢复、运维审计不够强；会把“认知图”误当“公司执行总线”。
- **不是纯 Temporal 立刻全迁**：可靠性最强，但会引入 worker/namespace/history/determinism/SDK/运维新复杂度；当前 OpenClaw 的 session/subagent/acp/browser/message 都需要先抽象适配，短期 ROI 不高。
- **不是纯 OpenClaw 原生长期不变**：短期最省，但状态仍分散在 run dir、session 文件、watcher、cron、消息回执里；规模一上来，跨流程追踪、补偿、回放、统一 SLA 会成为瓶颈。

### 建议目标架构
- **控制平面**：OpenClaw 原生（入口、路由、权限、人审、消息线程）
- **执行平面**：subagent / ACP / browser / message / cron / local jobs
- **状态平面**：统一 Task Registry + 状态机 + 事件总线 + 审计日志
- **耐久编排平面**：P2 起对关键流程接入 Temporal
- **认知编排平面**：LangGraph 仅用于单 agent 内部复杂推理/工具调用图

## A. 三层交付

## A1. 结论摘要

### 推荐决策
**推荐“OpenClaw Native+”到“Hybrid(Temporal selective)”的两段式路线。**

- **当前阶段主选**：OpenClaw 原生编排增强版
- **中期目标**：Hybrid（OpenClaw 控制平面 + Temporal 耐久工作流）
- **明确不推荐**：LangGraph 作为公司级总编排器

### 决策标准
按当前 OpenClaw 基线，最重要的不是理论能力上限，而是：
1. 是否兼容现有 `session/subagent/acp/watcher/cron/message/browser`
2. 是否能统一状态与终态语义
3. 是否能把失败恢复、回调、人工介入做成标准件
4. 是否能低风险渐进迁移

### 最终判断
- **若只看终极可靠性**：Temporal 最强
- **若只看当前落地效率**：OpenClaw 原生最强
- **若看“未来 6~12 个月的综合适配度”**：**Hybrid 最优**

## 1. OpenClaw 当前架构基线（作为评估前提）

## 1.1 当前已经存在的编排资产

### a) session / subagent
- 主会话负责接收需求、路由、验收
- 长任务通过 `sessions_spawn(runtime="subagent")` 派发
- subagent 本地执行依赖 runner + watcher + run dir 协议

这说明 OpenClaw 已经天然是 **“控制平面 + 子执行单元”** 的模式，而不是简单聊天机器人。

### b) ACP
- ACP 适用于外部异步系统、编码工具、人审/CI 类外部会话
- 当前 ACP 状态通过 session 文件与 bridge 机制回流

这说明 OpenClaw 已经有 **第二执行平面**，但和 subagent 不共用同一状态协议。

### c) watcher / cron / callback bus
- watcher 负责消费状态变化
- cron 负责周期性检查与补偿
- callback bus 负责回原频道/线程

这说明 OpenClaw 具备 **事件驱动雏形**，但当前更多是 script-first + file-first，而不是统一 durable workflow engine。

### d) message / browser
- `message` 是对外通知、线程回传、人审交互的关键副作用通道
- `browser` 是需要页面状态验证或页面操作的关键执行器

这意味着公司级编排必须把 **message/browser 当成标准 activity adapter**，而不是把所有流程都收敛成纯 API 工作流。

## 1.2 当前的核心问题

1. **状态分散**：subagent 在 run dir，ACP 在 session 文件，watcher 在 registry，delivery 在消息通道。
2. **重试口径不统一**：subagent 有 profile/stall 语义，ACP/browser/message/cron 的 retry 语义尚未完全统一。
3. **观测维度割裂**：当前能看“局部真值”，但不容易看到“一个公司级任务的完整 timeline”。
4. **人工介入缺统一协议**：有消息/线程能力，但缺标准化的 HITL 状态机。
5. **跨运行时恢复不足**：当前更像一组良好脚本，而不是统一的公司级 durable orchestration。

**所以这次选型的本质，不是选哪个框架更酷，而是选哪个框架最适合补 OpenClaw 现在的短板。**

## 2. 评估维度定义

| 维度 | 我们真正关心的工程问题 |
|---|---|
| 状态管理 | 是否有统一状态、恢复点、跨天持久化、回放能力 |
| 可靠性 / 重试 | 失败是否可分类、可重试、可补偿、可超时控制 |
| 可观测性 | 是否能看到 timeline、输入输出、重试轨迹、人工信号 |
| Human-in-the-loop | 是否能标准化等待人工、审批、修订、超时升级 |
| 多 agent 协作 | 是否能表达 main/subagent/acp/外部执行器之间的协同 |
| 与现有运行时兼容性 | 是否能低成本接住 session/subagent/acp/browser/message/cron |
| 实施复杂度 | 需要新增多少基础设施、SDK、worker、运维与心智负担 |
| 迁移成本 | 是否能增量落地，还是必须大爆炸迁移 |

## 3. 四个方案的工程判断

## 3.1 Temporal

### 适配判断
Temporal 最适合解决 OpenClaw 将来会越来越痛的几件事：
- 跨天/跨周长事务
- 重试/backoff/timer/signal
- durable history
- worker 崩溃后恢复
- 严格的活动边界与补偿语义

### 对 OpenClaw 的价值
如果未来出现这些流程，Temporal 很有价值：
- 多日内容审核/发布/回流闭环
- 外部 ACP 编码任务 + 人审 + 合并 + 回执
- 多市场定时扫描 + 条件触发 + 异常升级
- 需要强 SLA 的回调、补偿和审计

### 主要代价
1. 要把现有执行器都包装成 activity：subagent、ACP、browser、message、cron。
2. 要建立 workflow state 与 OpenClaw 线程/session 状态之间的映射。
3. 要接受 Temporal 的确定性/版本化约束，团队需要新方法论。
4. 要多一套 worker/queue/namespace/运维告警体系。

### 结论
**Temporal 适合作为“关键流程耐久引擎”，不适合短期直接取代 OpenClaw 原生控制平面。**

---

## 3.2 LangGraph

### 适配判断
LangGraph 强项在：
- 表达 agent 的节点图
- 维护图状态/checkpoint
- 人工断点、恢复、tool routing
- 复杂 LLM 推理流

### 它对 OpenClaw 真正有用的地方
- 某个 subagent 内部需要多步推理、检索、评审、反思、工具调用
- 一个复杂内容生成/审校代理需要图式控制而非单 prompt
- 人机协同流程需要在“单 agent 思维链”里插入人工校对

### 它不适合作为总编排器的原因
1. OpenClaw 的主问题不是“单 agent 内部图不够强”，而是“跨 runtime 的状态与恢复不统一”。
2. session/subagent/acp/browser/message/cron 这些都是现实执行器，不是 LangGraph 原生抽象对象。
3. LangGraph 即便能做 checkpoint，也不天然等价于公司级 durable execution、delivery idempotency、SLA 监控。
4. 如果把公司级执行总线做成 LangGraph，容易把工程可靠性问题重新包成 agent graph 问题。

### 结论
**LangGraph 更适合作为叶子层 agent runtime，而不是 OpenClaw 的公司级 orchestration backbone。**

---

## 3.3 OpenClaw 原生编排

### 当前优势
- 完全贴合现有运行时与工具模型
- 不需要新增外部基础设施
- 已有 runner/watcher/profile/milestone/terminal semantics
- 已有 callback bus / cron / adapter 思路
- 已有人审与消息线程的天然入口

### 当前上限
- 状态协议尚未彻底统一
- 观测以局部文件/脚本为主
- 补偿、回放、跨天恢复仍偏工程约定，不是耐久平台能力
- 流程复杂后，main 很容易继续承担过多 orchestration 认知负担

### 结论
**OpenClaw 原生编排是最好的“现在式起点”，但不是最好的“唯一终局”。**

---

## 3.4 Hybrid（推荐）

### 核心思想
把问题拆成两类，而不是试图让一个框架包打天下：

1. **控制平面问题**：入口、权限、线程、人审、agent 路由、工具治理  
   → 继续由 OpenClaw 负责
2. **耐久执行问题**：跨天重试、等待信号、严格超时、补偿、审计  
   → 关键流程接入 Temporal
3. **认知子流程问题**：复杂 agent 思考与工具路线图  
   → LangGraph 仅做叶子层

### 推荐判断
这是最接近 OpenClaw 当前现实的路线：
- 不浪费已有 session/subagent/acp/watcher/message/browser 资产
- 不把所有问题错塞给 LangGraph
- 也不在 P0 就把全公司执行链路搬进 Temporal

## 4. 推荐模块边界

## 4.1 控制平面（保留在 OpenClaw）

职责：
- 用户请求接入
- 任务拆分与路由
- 线程/频道上下文维护
- 人工确认/中止/升级决策
- 权限边界与工具策略

输入：用户消息、定时任务、外部事件  
输出：`orchestration_task`、人工信号、最终回复

## 4.2 状态平面（P0 就该补齐）

职责：
- 公司级 task registry
- 统一状态机
- 统一 terminal semantics
- event log / audit log
- delivery 去重键

不要再让 subagent/ACP/browser/message 各自定义“自己的完成语义”。

## 4.3 执行适配层

统一抽象为 `activity adapter`：
- `subagent-run`
- `acp-session`
- `browser-job`
- `message-delivery`
- `cron-trigger`
- `local-script`

每个 adapter 必须声明：
- 启动方法
- 心跳/状态来源
- 可重试性
- 幂等键
- 超时策略
- 人工介入点
- 终态映射规则

## 4.4 耐久编排层（P2 逐步引入 Temporal）

只接这类流程：
- 跨天等待
- 多次重试后仍需恢复
- 需要 timer/signal/compensation
- 需要严格审计与完整 timeline

**不要把所有轻量子任务都送进 Temporal。** 否则复杂度会反噬。

## 4.5 认知编排层（LangGraph 叶子化）

只在以下场景使用：
- 某个 agent 内部的多步推理图
- 需要 interrupt/checkpoint 的复杂审校流
- 复杂 tool routing 但不承担公司级 delivery/SLA

## 5. 建议事件流 / 状态机

## 5.1 统一状态机

建议公司级状态机：

```text
created
  -> accepted
  -> planned
  -> dispatched
  -> running
  -> waiting_external
  -> waiting_human
  -> retrying
  -> validating
  -> delivering
  -> completed / failed / timeout / cancelled / degraded
```

### 关键说明
- `waiting_external`：等 ACP、browser、cron、外部系统
- `waiting_human`：等老板/审核人/值班人确认
- `retrying`：标准 backoff 中，不应刷屏
- `degraded`：部分能力失败但系统给出可接受降级结果

## 5.2 统一事件模型

```text
TaskCreated
TaskPlanned
TaskDispatched
ActivityStarted
ActivityHeartbeat
MilestoneRaised
ExternalStateObserved
HumanSignalReceived
RetryScheduled
ValidationPassed
ValidationFailed
CallbackDispatched
CallbackDelivered
TaskCompleted
TaskFailed
TaskTimedOut
TaskCancelled
```

## 5.3 参考事件流

```text
用户消息
 -> OpenClaw 控制平面创建 orchestration_task
 -> 路由到 subagent / ACP / browser / cron
 -> adapter 上报 started / milestone / state changes
 -> watcher / event bus 更新 task registry
 -> 如需人工，发 message 并进入 waiting_human
 -> 收到人工信号后继续
 -> 进入 validating
 -> callback bus 投递最终结果
 -> closed / archived
```

## 5.4 runtime 状态映射建议

| Runtime 原生状态 | 公司级状态 |
|---|---|
| subagent `completed` | `completed` |
| subagent `timeout_stall` / `timeout_total` | `timeout` |
| subagent `process_exit` | `failed` |
| ACP `running` | `waiting_external` / `running` |
| ACP `completed` | `completed` |
| ACP `failed` / `cancelled` | `failed` / `cancelled` |
| browser 登录失效/DOM 不确定 | `waiting_human` 或 `degraded` |
| message 发送失败但可补发 | `delivering` / `retrying` |

## 6. 失败恢复设计

## 6.1 恢复策略按 adapter 分类，而不是一刀切

### subagent
- 保留现有 profile / idle / stall grace
- 把 `retryable` 与 `suggestedNextAction` 升级为 task policy 输入
- 同类失败最多自动重试 1 次，避免死循环

### ACP
- session 文件未出现：短暂视为 `running`，低置信度
- 长时间无更新：转 `timeout` 或 `waiting_human`
- 必须有 bridge，把 ACP 终态同步到统一 task registry

### browser
- 页面结构漂移：先重试一次 selector/DOM snapshot
- 登录失效：直接转 `waiting_human`
- 涉及副作用操作时必须幂等保护或人工确认

### message
- 发送失败：指数退避重试
- 连续失败：dead-letter + 升级 main
- 成功投递后严禁因 watcher 重扫而重复 final

### cron / watcher
- 必须支持 catch-up window，避免漏掉短窗口终态
- watcher 自己也要有 heartbeat 与 backlog 指标

## 6.2 补偿原则

- **可重试优先重试**
- **不可安全重试优先人工确认**
- **外部副作用必须幂等或可补偿**
- **最终回执只能有一个 authoritative final**

---

## 7. 质量门设计

## 7.1 P0 质量门

1. **状态契约已冻结**：统一 schema / state / event / terminal semantics
2. **幂等键可验证**：重复 watcher/重复投递不会重复 final
3. **端到端回放可验证**：至少能重放 task timeline
4. **失败分类清晰**：timeout / process_exit / cancelled / degraded 不能混淆
5. **人工断点可验证**：approve/reject/revise/timeout 都有测试

## 7.2 上线质量门

| Gate | 要求 |
|---|---|
| Gate 0 | schema + adapter contract review 通过 |
| Gate 1 | shadow mode：只记录不回调 |
| Gate 2 | canary：仅 1 条流程真实投递 |
| Gate 3 | 增加到 10~20% 流量 |
| Gate 4 | 默认启用，但保留 feature flag 回退 |

---

## 8. 回退方案

## 8.1 总体原则
- **任何新编排层都不能破坏 OpenClaw 当前可用路径**
- **新层失败时，入口仍可退回原生 sessions_spawn + watcher + callback**

## 8.2 Temporal 引入时的回退

1. 先 **dual write**：原生 task registry 与 Temporal workflow 同时记录
2. 先 **shadow execute**：Temporal 只观测/模拟，不真正发 final callback
3. 再 **per-workflow cutover**：按流程级别切换，不按全局切换
4. 若 Temporal worker/namespace 出问题：
   - 停止新建 Temporal workflow
   - 新任务回退到 OpenClaw Native+
   - 已在 Temporal 中运行的流程继续收尾，不强行搬迁中途状态

## 8.3 LangGraph 的回退

LangGraph 只允许出现在叶子层，因此回退很简单：
- 用直接 prompt / 线性 agent flow 替换该叶子
- 不影响公司级 task registry / callback / delivery 语义

## 9. P0 / P1 / P2 路线图

## P0（1~2 周）：先把 OpenClaw Native 变成“公司级可管控”

目标：**不引入新基础设施，先统一状态与观测。**

交付：
1. `orchestration_task` schema v1
2. unified state machine v1
3. subagent / ACP 状态映射器
4. callback 幂等与 delivery audit
5. task timeline viewer / dashboard 最小版

成功标准：
- 至少 1 条 subagent 流程 + 1 条 ACP 流程能在同一个 timeline 中看到全链路状态
- 重复 watcher 不会重复 final
- 人工确认能进入 `waiting_human` 再恢复

## P1（2~6 周）：扩展为真正的多执行器编排层

目标：**把 browser / cron / message 也纳入统一 activity adapter。**

交付：
1. browser adapter
2. cron adapter
3. message delivery adapter
4. dead-letter + escalation
5. retry policy registry
6. company-level observability board

成功标准：
- 3 类以上 runtime 能统一追踪
- 失败分类与补偿策略标准化
- 人工介入 SLA 有明确定义

## P2（6 周+）：引入 Temporal 处理高 SLA 长事务

目标：**只迁移真正值得迁移的流程。**

优先迁移对象：
- 跨天等待审批/回执的流程
- 需要 timer/signal/compensation 的流程
- 需要强审计、强恢复、强幂等的流程

不迁移对象：
- 纯短任务
- 一次性文档/分析型 subagent
- 简单 message/browser 调用
- 单 agent 内部思维流（给 LangGraph 或普通 agent prompt 即可）

成功标准：
- 至少 1 条关键流程在 Temporal 跑通，且不影响现有 OpenClaw 入口/回调体验
- 有清晰 cutover 与 rollback runbook

## 10. 最终建议（给 CTO 的决策口径）

### 推荐口径
如果老板现在要一个可执行决策，我建议直接定成：

> **公司级总编排仍以 OpenClaw 为主控；P0/P1 先把状态层、事件总线、适配器层标准化；P2 只把高价值长事务接入 Temporal；LangGraph 不做公司级总编排，只做 agent 内部复杂认知子图。**

### 这样做的好处
- **短期不打断现有生产路径**
- **中期补上最真实的工程短板：状态、重试、审计、HITL**
- **长期保留引入 Temporal 的上限**
- **避免把 LangGraph 用错位，导致认知编排和执行编排混在一起**

### 最关键的一句 why-not
- **不是 LangGraph-first**，因为我们缺的不是 agent 图，而是 durable execution。  
- **不是 Temporal-first 全迁**，因为我们现在缺的首先是统一抽象与状态协议。  
- **不是 pure native forever**，因为公司级异步流程最终一定会撞上恢复、审计、补偿和 SLA 的天花板。
